## 一、核心质量

### 1.1 description 具体性

- description 必须具体，包含触发关键词
- 必须同时包含 WHAT（功能描述）和 WHEN（触发时机）
- 必须用第三人称编写（如"处理 Excel 文件"而非"我可以帮你处理"）
- "何时使用"信息必须在 description 中，而非正文中（正文仅在触发后加载）

正面示例:
```
description: 从 PDF 文件中提取文本和表格、填充表单、合并文档。在处理 PDF 文件或用户提及 PDF、表单或文档提取时使用。
```

反面示例:
```
description: 帮助处理文档
```

### 1.2 命名规范

- name 最多 64 字符
- 仅含小写字母、数字、连字符
- 不含 XML 标签
- 不含保留词 "anthropic"、"claude"
- 推荐动名词形式: `processing-pdfs`、`analyzing-spreadsheets`
- 避免模糊名称: `helper`、`utils`、`tools`

### 1.3 SKILL.md 正文行数

- 正文必须 < 500 行
- 接近限制时应拆分到独立参考文件
- 使用渐进式披露模式管理内容

### 1.4 渐进式披露

- SKILL.md 作为概览，指向详细材料
- 参考文件仅在 Claude 需要时加载
- 三种内容类型: 指令(markdown)、代码(scripts)、资源(assets)

### 1.5 术语一致性

- 全文使用统一术语，不混用同义词
- 正面: 始终使用 "API 端点"
- 反面: 混用 "API 端点"、"URL"、"路由"、"路径"

### 1.6 无时间敏感信息

- 不包含会过期的日期性表述
- 如需保留旧信息，使用"旧模式"折叠部分

反面示例: "如果你在 2025 年 8 月前使用，请用旧 API"

### 1.7 示例具体性

- 示例必须具体，提供输入/输出对
- 不使用抽象或笼统的描述

### 1.8 引用文件深度

- 引用文件必须从 SKILL.md 直接链接（一级深度）
- 不允许从引用文件再链接到其他引用文件

### 1.9 参考文件结构

- 超过 100 行的参考文件必须包含目录
- 确保 Claude 即使部分读取也能看到完整范围

### 1.10 自由度匹配

- 高自由度(文本指令): 多种方法有效、上下文决定
- 中自由度(伪代码/模板): 存在首选模式、允许变化
- 低自由度(具体脚本): 操作脆弱、一致性关键

### 1.11 工作流清晰度

- 复杂任务分解为清晰步骤
- 提供可跟踪的检查清单
- 包含条件分支指导

## 二、代码和脚本

### 2.1 脚本解决问题而非推卸

- 脚本应处理错误条件，而非让 Claude 猜测
- 提供明确的错误处理和备选方案

正面示例:
```python
def process_file(path):
    try:
        with open(path) as f:
            return f.read()
    except FileNotFoundError:
        print(f"文件 {path} 未找到，创建默认值")
        with open(path, 'w') as f:
            f.write('')
        return ''
```

### 2.2 错误处理明确

- 每种异常有对应的处理策略
- 提供有意义的错误消息

### 2.3 无魔法数字

- 所有常数都有注释说明其存在的理由
- 反面示例: `TIMEOUT = 47  # 为什么是 47？`

### 2.4 包依赖列出

- 所有依赖包在说明中明确列出
- 验证包在目标运行环境中可用

### 2.5 无 Windows 风格路径

- 所有文件路径使用正斜杠 `/`
- 正面: `scripts/helper.py`
- 反面: `scripts\helper.py`

### 2.6 验证/验证步骤

- 关键操作提供验证脚本
- 使用"计划-验证-执行"模式处理批量操作

### 2.7 反馈循环

- 质量关键任务包含验证循环: 运行验证 -> 修复错误 -> 重复
- 明确标注"仅在验证通过时才继续"

### 2.8 MCP 工具引用

- 使用全限定名: `ServerName:tool_name`
- 示例: `BigQuery:bigquery_schema`
- 不使用服务器前缀会导致 "找不到工具" 错误

## 三、测试与迭代

### 3.1 评估场景

- 至少创建 3 个评估场景
- 先建评估再写大量文档（评估驱动开发）

### 3.2 多模型测试

- 使用 Haiku、Sonnet、Opus 全部测试
- Opus 上完美的内容可能需要为 Haiku 提供更多细节

### 3.3 真实场景测试

- 使用真实使用场景而非测试场景
- 观察 Claude 实际如何导航 skill（读取顺序、是否遗漏文件）

### 3.4 团队反馈

- 与团队分享 skill 并收集反馈
- 关注: 是否在预期时激活、说明是否清晰、是否有缺失

---

## 不应包含的文件

Skill 目录中不应出现以下文件:
- README.md
- INSTALLATION_GUIDE.md
- QUICK_REFERENCE.md
- CHANGELOG.md
- 任何与 skill 功能无直接关系的辅助文档
